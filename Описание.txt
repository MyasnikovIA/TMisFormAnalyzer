# Инструкция пользователя по работе с анализатором форм T-MIS

## 1. Общее описание программы

**Анализатор форм T-MIS (M2/D3)** - это инструмент для анализа XML-форм системы T-MIS. Программа позволяет:
- Извлекать SQL запросы из форм
- Определять используемые таблицы, вьюхи, пакеты и функции
- Анализировать зависимости между формами
- Генерировать детальные отчеты

## 2. Системные требования

### Минимальные требования:
- **Операционная система**: Windows 10/11, Linux (Ubuntu 18.04+), macOS 10.15+
- **Java**: версия 11 или выше
- **Оперативная память**: минимум 2 GB
- **Свободное место на диске**: минимум 500 MB

### Доступ к базам данных (опционально):
- Oracle Database (для анализа вьюх)
- PostgreSQL (для анализа вьюх)

## 3. Установка и запуск

### 3.1. Установка
1. Распакуйте архив с программой в любую папку
2. Убедитесь, что установлена Java 11 или выше (проверить: `java -version` в командной строке)

### 3.2. Запуск
**Способ 1 (графический интерфейс):**
```bash
java -jar TMisFormAnalyzer.jar
```

**Способ 2 (с указанием пути к проекту):**
```bash
java -jar TMisFormAnalyzer.jar --project-path /путь/к/mis
```

## 4. Первоначальная настройка

### 4.1. Открытие настроек
Нажмите кнопку **"Настройки"** в верхней панели

### 4.2. Вкладка "Подключения к БД"

#### Обязательные настройки:
| Поле | Описание | Пример |
|------|----------|--------|
| **Путь к проекту T-MIS** | Корневая директория проекта T-MIS | `/var/www/t-mis/mis` |
| **Каталог для отчетов** | Куда сохранять отчеты | `SQL_info` |

#### Настройки Oracle (опционально):
| Поле | Описание | Пример |
|------|----------|--------|
| **Oracle URL** | Строка подключения к Oracle | `jdbc:oracle:thin:@192.168.241.141:1521/med2dev` |
| **Oracle User** | Имя пользователя | `dev` |
| **Oracle Password** | Пароль | `*****` |

#### Настройки PostgreSQL (опционально):
| Поле | Описание | Пример |
|------|----------|--------|
| **PostgreSQL URL** | Строка подключения к PostgreSQL | `jdbc:postgresql://192.168.241.137:5432/med2dev` |
| **PostgreSQL User** | Имя пользователя | `postgres` |
| **PostgreSQL Password** | Пароль | `*****` |

#### Проверка подключения:
- Нажмите кнопку **"Проверить"** рядом с каждым подключением
- Зеленый статус "✅ Подключение успешно" означает, что все работает
- Красный статус с ошибкой - проверьте правильность введенных данных

### 4.3. Вкладка "Настройки отчета"

#### Быстрые настройки:
| Режим | Описание |
|-------|----------|
| **Минимальный** | Только базовая структура (быстро, но мало информации) |
| **Стандартный** | Рекомендуемый режим (оптимальный баланс) |
| **Полный** | Максимально подробный отчет (долго) |

#### Детальные настройки:

| Опция | Описание | Когда включать |
|-------|----------|----------------|
| **Показывать SQL запросы** | Выводит полный текст SQL | Для детального анализа запросов |
| **Список вызываемых форм в JS** | Показывает формы, вызываемые из JavaScript | Для анализа связей между формами |
| **Используемые таблицы и вьюхи** | Список всех D_* и D_V_* объектов | Всегда полезно |
| **Таблицы через вьюхи** | Анализирует вьюхи и показывает таблицы внутри них | Требует доступа к БД |
| **Композиции из JS** | Извлекает UnitEdit композиции из JS | Для анализа структуры форм |
| **Детальное содержимое вьюх** | Полный разбор каждой вьюхи | Для глубокого анализа БД |

## 5. Работа с программой

### 5.1. Ввод списка форм

Список форм можно вводить в разных форматах:

**Формат 1 (полный путь):**
```
/Forms/HospDirAccounting/hospdiraccounting.frm
/Forms/HospPlan/hp_directions_edit.frm
```

**Формат 2 (относительный путь с расширением):**
```
HospDirAccounting/hospdiraccounting.frm
HospPlan/hp_directions_edit.frm
```

**Формат 3 (относительный путь без расширения):**
```
HospDirAccounting/hospdiraccounting
HospPlan/hp_directions_edit
```

**UserForms пути:**
```
UserFormsSaratov/ARMMainDoc/stac_pat_in_hpk_plan.frm
UserFormsEgis3Tomsk/ARMMainDoc/neotl_patients.d/neotl_patients.dfrm
```

**Особенности:**
- Каждая форма на новой строке
- Строки, начинающиеся с `#`, считаются комментариями
- Если список пуст - анализируется весь проект

### 5.2. Запуск анализа

1. Убедитесь, что список форм заполнен или оставлен пустым (для анализа всего проекта)
2. Нажмите кнопку **"Запуск анализа"**
3. Наблюдайте за процессом в логе и прогресс-баре

### 5.3. Управление процессом

| Действие | Кнопка | Описание |
|----------|--------|----------|
| Запуск | **Запуск анализа** | Начинает анализ форм |
| Остановка | **Остановка** | Прерывает выполнение (может занять несколько секунд) |

### 5.4. Просмотр результатов

После завершения анализа:
1. Нажмите **"Открыть директорию отчета"** для просмотра результатов
2. Отчеты сохраняются в папке, указанной в настройках (по умолчанию `SQL_info/`)

## 6. Структура отчетов

### 6.1. Основные файлы отчетов

| Файл | Описание |
|------|----------|
| `forms_report.txt` | Основной отчет по формам |
| `forms_report_all.txt` | Полный отчет (все формы, включая SQL) |
| `forms_report_all_without_sql.txt` | Отчет без SQL запросов |
| `tables_views_report.txt` | Список всех таблиц и вьюх |
| `packages_functions_report.txt` | Список всех пакетов и функций |
| `summary_report.txt` | Сводная статистика |
| `view_dependencies_report.txt` | Зависимости вьюх от таблиц |

### 6.2. Содержимое отчета по форме

```
ФОРМА: HospDirAccounting/hospdiraccounting.frm
----------------------------------------------------------------------------------------------------
Базовая форма: /путь/к/форме.frm
СТАТУС: БАЗОВАЯ ФОРМА (без переопределений)

ЮЗЕРФОРМЫ:
     ===== UserFormsRegion =====
        ПОЛНАЯ ЗАМЕНА: /путь/к/замене.frm

Список подключаемых форм subForm:
     Visit/PrintVisit
     HospDirAccounting/search_patient

Список вызываемых форм в JS:
     Persmedcard/persmedcard_edit.frm
     DiseaseCase/diseasecase.frm

SQL ЗАПРОСЫ (2):
     (содержимое SQL запросов)

ИСПОЛЬЗУЕМЫЕ ВЬЮХИ:
     D_V_HOSP_HISTORIES
     D_V_VISITS

ТАБЛИЦЫ, ИСПОЛЬЗУЕМЫЕ ЧЕРЕЗ ВЬЮХИ (уникальные для этой формы):
     D_HOSP_HISTORIES
     D_VISITS

ИСПОЛЬЗУЕМЫЕ ПАКЕТЫ И ФУНКЦИИ:
     D_PKG_HOSP_HISTORIES.GET_STATUS

СИСТЕМНЫЕ ОПЦИИ:
     AllowSelectAllLpu;
     BedTypeDefault;

КОНСТАНТЫ:
     JOURNAL_PCR_COVID;

РАЗОБРАТЬ АНАЛИТИКОМ:
     D_UNKNOWN_OBJECT
```

### 6.3. Пояснение статусов форм

| Статус | Описание |
|--------|----------|
| **БАЗОВАЯ ФОРМА** | Форма из каталога Forms, без переопределений |
| **ПОЛНОСТЬЮ ЗАМЕНЕНА** | Форма полностью переопределена в UserForms |
| **ЧАСТИЧНО ПЕРЕОПРЕДЕЛЕНА** | Имеются переопределения через .d каталог или .dfrm файлы |

## 7. Работа с логами

### 7.1. Просмотр лога
- Лог отображается в правой части окна
- Каждая запись содержит временную метку
- Цветовая индикация: зеленый текст на темном фоне

### 7.2. Управление логом
- **"Очистить лог"** - удаляет все сообщения
- **"Сохранить лог"** - сохраняет текущий лог в отдельный файл

## 8. Решение типовых проблем

### 8.1. Ошибка подключения к Oracle

**Ошибка**: `The Network Adapter could not establish the connection`

**Решение:**
1. Проверьте доступность сервера: `ping 192.168.241.141`
2. Проверьте порт: `telnet 192.168.241.141 1521`
3. Отключите временно брандмауэр Windows
4. Убедитесь, что VPN подключен (если требуется)

### 8.2. Ошибка подключения к PostgreSQL

**Ошибка**: `Connection refused`

**Решение:**
1. Проверьте доступность сервера: `ping 192.168.241.137`
2. Проверьте порт: `telnet 192.168.241.137 5432`
3. Проверьте настройки pg_hba.conf на сервере

### 8.3. Форма не найдена

**Ошибка**: `Базовая форма не найдена`

**Решение:**
1. Проверьте правильность пути в списке форм
2. Убедитесь, что проект T-MIS доступен по указанному пути
3. Проверьте наличие файла `.frm` в каталоге Forms

### 8.4. Нет таблиц в отчете

**Проблема**: В отчете отображаются только вьюхи, но нет таблиц

**Решение:**
1. Включите настройку **"Таблицы через вьюхи"**
2. Убедитесь, что подключение к Oracle настроено правильно
3. Проверьте, что вьюхи существуют в БД

### 8.5. Приложение зависло при остановке

**Решение:**
1. Подождите 10-15 секунд
2. Закройте приложение через Task Manager (Windows) или kill (Linux)
3. Запустите приложение заново

## 9. Советы по эффективному использованию

### 9.1. Для быстрого анализа:
- Включите только **"Используемые таблицы и вьюхи"**
- Выключите **"Показывать SQL запросы"**
- Используйте **Стандартный** пресет

### 9.2. Для глубокого анализа:
- Включите все опции
- Используйте **Полный** пресет
- Убедитесь, что подключения к БД работают

### 9.3. Для анализа отдельных форм:
- Введите конкретные пути форм
- Не включайте анализ всего проекта

### 9.4. Для поиска зависимостей:
- Включите **"Список вызываемых форм в JS"**
- Включите **"Список подключаемых форм subForm"**

## 10. Часто задаваемые вопросы (FAQ)

### Вопрос 1: Нужно ли подключаться к БД для работы?
**Ответ**: Не обязательно. Без подключения к БД вы получите список вьюх, но таблицы внутри вьюх не будут проанализированы.

### Вопрос 2: Как долго длится анализ?
**Ответ**:
- 1 форма: 1-5 секунд
- 100 форм: 1-2 минуты
- 1000 форм: 10-20 минут
- Весь проект (2000+ форм): 30-60 минут

### Вопрос 3: Можно ли анализировать формы в фоне?
**Ответ**: Да, анализ выполняется в отдельном потоке, вы можете свернуть приложение.

### Вопрос 4: Сохраняются ли настройки после перезапуска?
**Ответ**: Да, все настройки (подключения, настройки отчета, список форм, позиция окна) автоматически сохраняются.

### Вопрос 5: Что делать, если отчет не содержит ожидаемые данные?
**Ответ**:
1. Проверьте, что форма действительно содержит SQL запросы
2. Убедитесь, что файл формы читается (проверьте права доступа)
3. Включите **"Показывать SQL запросы"** для отладки
4. Проверьте лог на наличие ошибок

## 11. Контакты поддержки

При возникновении проблем:
1. Сохраните лог программы (кнопка **"Сохранить лог"**)
2. Сделайте скриншот окна с ошибкой
3. Опишите последовательность действий, приведших к ошибке

## 12. Глоссарий

| Термин | Описание |
|--------|----------|
| **Вьюха** | Представление в базе данных (VIEW), начинается с D_V_ |
| **Форма** | XML-файл с расширением .frm или .dfrm |
| **UserForms** | Каталог с переопределениями форм |
| **SubForm** | Подформа, встраиваемая в основную форму |
| **DataSet** | Компонент, содержащий SQL запрос для получения данных |
| **Action** | Компонент, содержащий SQL запрос для выполнения действий |
| **Кэш** | Временное хранилище данных для ускорения работы |
| **Пакет** | PL/SQL пакет в Oracle, начинается с D_PKG_ |
